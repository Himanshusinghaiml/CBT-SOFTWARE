Abstract

CBT Software — Computer Based Testing System

The CBT (Computer Based Testing) Software is a full-stack web application designed to digitize and streamline the entire lifecycle of computer-based examinations. The system provides a multi-role platform supporting four user types: Super Admin, Company (exam organizer), Exam Center, and Student.

Built using the Python Django framework with an SQLite3 database and Bootstrap 5 frontend, the system enables exam companies to create and manage tests, build a question bank (including AI-powered automatic question generation via the OpenAI API), and register multiple exam centers with secure hashed credentials.

Exam centers log in to manage student entry, seating arrangements, and emergency procedures. Students take fully timed Multiple Choice Question (MCQ) exams through an AJAX-driven interface that supports question navigation, review marking, and auto-submission on timer expiry.

The system implements session-based authentication, PBKDF2-SHA256 password hashing, Django REST Framework for API endpoints, and an optional OpenAI integration for automated question generation categorized by topic and difficulty level.

Keywords: Computer Based Testing, Django, MCQ Examination, AI Question Generation, Exam Management System, Online Examination, Web Application.

Chapter 1: Introduction

1.1 Problem Statement

Traditional paper-based examinations are associated with significant challenges: high printing and logistics costs, susceptibility to paper leaks, manual and error-prone evaluation, slow result declaration, limited scalability, and large administrative overhead. These problems are especially acute for organizations that conduct examinations at multiple geographically distributed centers simultaneously.

There is a clear need for a reliable, scalable, and secure digital platform that automates question bank management, examination delivery, multi-center coordination, and result processing — while remaining easy to deploy and maintain by non-technical administrators.

1.2 Objectives

  • Design and develop a multi-role web-based examination management system.
  • Provide a secure company portal for creating tests, managing question banks, and registering exam centers.
  • Provide a center portal for managing student entry and seat arrangements.
  • Deliver an AJAX-driven student exam interface with countdown timer, question palette, and review marking.
  • Implement secure password storage using PBKDF2-SHA256 hashing.
  • Integrate an AI-powered question generation module using the OpenAI API.
  • Build RESTful API endpoints using Django REST Framework.
  • Ensure role-based access control through custom session management.

1.3 Scope of the Project

The CBT Software covers the following scope:

  • In scope: MCQ-based examinations, multi-center management, question bank with AI generation, timed exam interface, session-based security, Django admin panel.
  • Out of scope: Video proctoring, descriptive answer evaluation, SMS/email notifications, payment gateway (these are planned as future enhancements).

Chapter 2: Literature Survey

2.1 Existing Systems

2.1.1 Google Forms / Microsoft Forms

Cloud-based quiz tools that allow basic MCQ creation and automatic grading. However, they lack multi-center management, exam-center-level security, a dedicated student interface with question palettes, or integration with an organizational exam workflow. They are not suitable for formal, proctored examinations at scale.

2.1.2 Moodle (LMS)

An open-source Learning Management System supporting online quizzes. While feature-rich, Moodle requires complex installation, has a steep learning curve for administrators, and is primarily designed for course management rather than standalone examination delivery.

2.1.3 HackerRank / TestGorilla

Commercial SaaS platforms for technical assessments. They are proprietary, subscription-based, and not customizable for local institutional requirements such as custom center management or government examination workflows.

2.1.4 Conducted Research

Several papers (Nwaocha, V.O. — 2012; Jimoh, R.G. et al. — 2012) highlight that computer-based testing reduces examination malpractice, speeds up result processing, and improves test validity compared to paper-based methods. AI-assisted question generation has been explored in recent works (Kurdi et al., 2020), demonstrating that NLP-based question generation significantly reduces content creation time for educators.

3.6 Module Description

The system is divided into five Django application modules:

Module 1: general_zone (Public & Authentication)

  • Serves the public-facing website (homepage, about, services, courses)
  • Handles Company signup and login (with PBKDF2 password hashing)
  • Handles Center login and Admin login
  • Provides Contact/Request Demo form
  • Exposes Company REST API via Django REST Framework

Module 2: companylogin (Company Portal)

  • Company dashboard with quick-access cards
  • Test management: create, list (ongoing/completed), delete
  • Center management: add, list, delete
  • Question bank: add manually, list with filters, delete, AI generation
  • Session-secured views using request.session['com_id']

Module 3: centerlogin (Center Portal)

  • Center dashboard
  • Student entry login interface
  • Student list management
  • Seat arrangement visualization
  • Emergency management page

Module 4: studentexam (Exam Interface)

  • Student home with login form
  • Instructions page with exam rules
  • AJAX-powered running exam interface with countdown timer
  • Question navigation (Next/Previous), Mark for Review, Clear Response
  • Question palette showing answered/unanswered/reviewed status
  • Submit confirmation page

Module 5: admin_zone (Super Admin)

  • Super admin management panel
  • Stores Company, Admin, Center list, and Contact data models
  • Accessible via Django's built-in /admin/ interface

4.3 Module Implementation

4.3.1 Authentication Flow

The system uses custom session-based authentication rather than Django's built-in authentication system. On successful login, the user's primary key is stored in the session: 

request.session['com_id'] = user.id      # Company login
request.session['center_id'] = user.id   # Center login

Every protected view checks for this session key at the start: 

if not request.session.get('com_id'):
    return redirect('homepage')

Passwords are hashed on registration using Django's make_password() which applies PBKDF2-HMAC-SHA256 with a random salt. Verification uses check_password(raw, hash).

4.3.2 Test Management

A Test model stores the exam metadata. Tests are created via a POST form, listed on the ongoing/completed pages based on their status field, and deleted via a dedicated view that redirects based on the test's previous status.

4.3.3 AJAX Exam Interface

The running exam page loads questions dynamically via AJAX calls to two JSON endpoints: /get-next-question/<id>/ and /get-previous-question/<id>/. The countdown timer is implemented in JavaScript using setInterval: 

const timer = setInterval(() => {
  const now = new Date().getTime();
  const distance = endTime - now;
  if (distance <= 0) {
    clearInterval(timer);
    document.querySelector('form').submit();
  }
  // update display...
}, 1000);

4.4 AI Question Generation

When an OPENAI_API_KEY is configured, company users can auto-generate questions by submitting a topic, count, and difficulty level. The system sends a structured prompt to OpenAI's gpt-3.5-turbo model and parses the response: 

prompt = f"Generate {count} MCQs on '{topic}' at {difficulty} difficulty.
Provide each with options A, B, C, D and the correct letter."

The parsed questions are saved to the Question model with generated_by='ai' and are immediately available in the question bank. This feature is fully optional — the system functions normally without it.

4.5 Security Implementation

  • Password Hashing: PBKDF2-SHA256 via Django's make_password()
  • CSRF Protection: Django CSRF middleware enabled for all POST forms ({% csrf_token %})
  • Session Security: Custom session key checking on every protected view
  • No Raw Passwords Displayed: Center passwords shown as ●●●●●●●● in UI
  • Input Validation: HTML5 required attributes and server-side Django ORM queries
  • SQL Injection Prevention: All database access via Django ORM (parameterized queries)
  • XSS Prevention: Django template auto-escaping enabled by default

Chapter 5: Testing

5.1 Testing Methodology

The project was tested using a combination of manual functional testing (black-box), integration testing across modules, and boundary value analysis. Django's built-in test framework was used for unit tests.

Testing Types Applied:

  • Unit Testing: Individual view functions tested for correct HTTP responses
  • Integration Testing: Login → Dashboard → Test Creation → Question Addition flow verified end-to-end
  • Boundary Testing: Empty form submissions, invalid credentials, non-existent question IDs
  • Security Testing: Session hijacking attempts, unauthorized URL access, CSRF token validation
  • UI Testing: Responsive layout on desktop and mobile browsers

Chapter 6: Conclusion and Future Work

6.1 Conclusion

The CBT Software successfully delivers a complete, self-contained Computer Based Testing platform that addresses the core challenges of paper-based examinations. The system implements the full examination lifecycle: from company registration and test/question creation, through multi-center management, to a secure timed MCQ examination interface for students.

Key accomplishments of this project include:

  • A clean multi-role Django web application serving four distinct user types
  • Secure authentication using PBKDF2-SHA256 password hashing and custom session management
  • A responsive, modern dark-themed administrative dashboard
  • An AJAX-driven timed exam interface with question palette, review marking, and auto-submission
  • An AI-powered question generation module using the OpenAI API
  • RESTful API endpoints using Django REST Framework
  • Comprehensive CRUD operations for all core entities (Tests, Centers, Questions)

The project demonstrates the practical application of full-stack web development principles, database design, security best practices, and modern AI API integration within a real-world examination management context.

6.2 Future Enhancements

  • Online Proctoring: Integration of webcam-based monitoring and AI-powered suspicious activity detection during exams.
  • Automated Result Calculation: Immediate score computation after exam submission and a student results dashboard.
  • Multi-format Questions: Support for True/False, fill-in-the-blank, and descriptive (essay) question types with manual grading.
  • Email Notifications: Automated email alerts for exam schedules, center credentials, and result announcements.
  • Advanced Analytics: Company-level dashboards showing question difficulty analysis, student performance graphs, and topic-wise breakdowns.
  • Mobile App: React Native or Flutter mobile application for exam delivery on tablets.
  • Production Deployment: Dockerized deployment with Gunicorn, Nginx, and PostgreSQL for production-grade reliability.
  • Question Negative Marking: Configurable negative marking per question for competitive examination formats.
